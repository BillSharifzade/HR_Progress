    1. Общая информация
Название модуля: Assessment (Ассессмент)
Назначение модуля: Модуль предназначен для проведения оценки сотрудников по утвержденным критериям компетенций с участием руководителей и асессоров.
Результаты ассессмента используются для:
    • формирования профиля сотрудника;
    • передачи данных в Матрицу компетенций (МК);
    • определения зон развития;
    • дальнейшего формирования ПИР.
    2. Основная логика модуля
Ассессмент запускается администратором.
Администратор:
    • определяет участников оценки;
    • определяет критерии оценки;
    • назначает асессоров;
    • устанавливает даты проведения.
После назначения ассессмента:
    • руководитель отдела получает форму оценки сотрудника;
    • руководитель департамента получает форму оценки сотрудника;
    • назначенные асессоры получают форму оценки сотрудника.
После завершения оценки:
    • система рассчитывает итоговые показатели;
    • результаты направляются администратору на проверку;
    • после подтверждения результаты публикуются в профиле сотрудника.

При выставлении оценки по каждой компетенции система должна автоматически отображать пользователю текстовую интерпретацию выбранного балла.

    • Руководитель отдела; 
    • Руководитель департамента; 
    • Асессор. 
7.2.1 Логика работы
После открытия формы оценки пользователь выбирает компетенцию и выставляет балл по шкале от 1 до 10.
После выбора оценки система автоматически определяет:
    • департамент сотрудника; 
    • грейд сотрудника; 
    • оцениваемую компетенцию; 
    • выбранный балл. 
На основании этих данных система выполняет поиск соответствующей текстовой интерпретации по следующему алгоритму:
Департамент → Грейд → Компетенция → Балл → Текстовая интерпретация
После успешного поиска система автоматически отображает текстовую интерпретацию рядом с выбранной оценкой.
7.2.2 Редактирование интерпретации
Пользователь, выставляющий оценку, должен иметь возможность:
    • оставить предложенную системой интерпретацию без изменений; 
    • дополнить автоматически сформированный текст; 
    • полностью заменить предложенную интерпретацию собственным комментарием. 
При сохранении оценки система должна сохранять:
    • итоговый балл; 
    • автоматически сформированную интерпретацию; 
    • итоговую редакцию текста пользователя. 
7.2.3 Справочник текстовых интерпретаций
Текстовые интерпретации должны храниться в отдельном справочнике системы.
Каждая запись справочника должна содержать:
    • Департамент; 
    • Грейд; 
    • Компетенцию; 
    • Балл (от 1 до 10); 
    • Текстовую интерпретацию; 
    • Версию записи; 
    • Дату изменения; 
    • Автора изменения. 
7.2.4 Управление справочником
Администратор должен иметь возможность:
    • создавать новые текстовые интерпретации; 
    • редактировать существующие; 
    • удалять неактуальные записи; 
    • копировать интерпретации между департаментами; 
    • копировать интерпретации между грейдами; 
    • просматривать историю изменений. 
Изменение справочника должно выполняться без участия разработчиков.
7.2.5 Использование интерпретаций
Автоматически сформированная интерпретация используется для:
    • стандартизации процесса оценки сотрудников; 
    • снижения субъективности при выставлении оценок; 
    • предоставления качественной обратной связи сотруднику; 
    • формирования итогового профиля компетенций; 
    • дальнейшего формирования Матрицы компетенций; 
    • подготовки рекомендаций для разработки ПИР. 
7.2.6 Исключительные ситуации
Если для выбранной комбинации:
    • департамент; 
    • грейд; 
    • компетенция; 
    • балл, 
    • в справочнике отсутствует текстовая интерпретация, система должна вывести уведомление:
«Для выбранной оценки текстовая интерпретация не настроена.»
При этом пользователь сохраняет возможность выставить оценку и оставить собственный комментарий.

    1. Источник данных: 
Для формирования групп используются только опубликованные и подтвержденные результаты ассессмента. В расчётах участвуют исключительно оценки асессоров.
Оценки руководителя отдела и руководителя департамента:
    • сохраняются в системе;
    • доступны администратору для анализа;
    • используются как дополнительная информация при принятии решений;
    • не участвуют в автоматическом ранжировании и формировании групп обучения.
    2. Расчет итогового балла для группировки
Для каждого сотрудника система должна рассчитать итоговый средний балл на основании оценок асессоров. Расчет выполняется как среднее арифметическое всех оценок асессоров по всем компетенциям. Допускаются дробные значения. 
После расчета система автоматически определяет:
    • минимальный итоговый балл среди всех участников;
    • максимальный итоговый балл среди всех участников.
    3. Ранжирование участников
После расчета итоговых баллов система должна автоматически отсортировать всех участников ассессмента по возрастанию итогового балла.
Пример:
1.02
1.18
1.37
9.85
Ранжирование выполняется автоматически после завершения и подтверждения кампании ассессмента.
    4. Автоматическое формирование групп
После ранжирования система должна автоматически сформировать группы обучения. 
Правила формирования:
    • размер группы по умолчанию составляет 12 человек;
    • размер группы должен быть настраиваемым администратором;
    • сотрудники распределяются строго последовательно по рейтингу;
    • в первую группу попадают сотрудники с самыми низкими итоговыми баллами;
    • во вторую группу попадают следующие сотрудники по рейтингу;
    • распределение продолжается до формирования групп для всех участников.
Если количество сотрудников не кратно размеру группы, последняя группа может содержать меньше участников.
    5. Анализ компетенций внутри группы
После формирования каждой группы система должна выполнить анализ результатов участников группы.
Для каждой компетенции рассчитывается:
    • средний балл по группе;
    • минимальный балл по группе;
    • максимальный балл по группе.
    6. Определение сильной стороны группы
Система должна определить одну компетенцию с самым высоким средним значением среди всех компетенций группы.
Результат отображается как: Сильная сторона группы
Пример: Коммуникация — 8.3
    7. Определение зон развития группы
Система должна определить от двух до трех компетенций с самыми низкими средними значениями в группе.
Результат отображается как:
Приоритетные зоны развития
Пример:
    • Лидерство - 4.8
    • Управление задачами - 5.1
    • Аналитическое мышление - 5.3
    8. Формирование рекомендаций
На основании анализа группы система должна автоматически подготовить рекомендации для AtS и администратора.
Для каждой группы система должна сформировать:
    • номер группы;
    • количество участников;
    • диапазон итоговых баллов группы;
    • список участников;
    • сильную компетенцию группы;
    • 2–3 приоритетные зоны развития группы.
    9. Отображение результатов
Результаты группировки отображаются в отдельном разделе:
«Группы обучения»

Для каждой группы отображается:
    • номер группы;
    • диапазон итоговых баллов;
    • количество участников;
    • список участников;
    • сильная компетенция;
    • зоны развития;
    • дата формирования группы.

Сформированные группы используются как основание для:

    • назначения внутренних программ развития.
    11. Действия администратора
Администратор должен иметь возможность:
    • просматривать автоматически сформированные группы;
    • изменять состав групп вручную;
    • переносить сотрудников между группами;
    • изменять размер группы;
    • повторно запускать формирование групп;
    • подтверждать итоговое распределение.
Все ручные изменения должны фиксироваться в журнале изменений.
    12. Критические правила
    1. Формирование групп выполняется только после подтверждения результатов ассессмента администратором.
    2. В расчетах (для формирования групп) используются только оценки асессоров.
    3. Один сотрудник может входить только в одну группу обучения.
    4. Ранжирование выполняется по возрастанию итогового среднего балла.
    5. При анализе используются только опубликованные результаты ассессмента.
    6. Размер группы по умолчанию составляет 12 человек и может изменяться администратором.
    7. Система должна автоматически пересчитывать группы после каждого нового цикла ассессмента.
    8. Результаты группировки являются рекомендательными и могут быть скорректированы администратором вручную.
    9. Все изменения состава групп должны сохраняться в журнале действий.
    5. Статусы кампании   ❌ НЕ ПОКРЫТО
> В assessment_periods нет поля статуса. Жизненный цикл (Черновик → Назначена → В процессе →
>      На проверке → Подтверждена → Опубликована) не реализован; есть лишь флаг is_active.
Кампания ассессмента должна поддерживать следующие статусы:
Черновик: Кампания создана, но не назначена участникам.
Назначена: Участники, руководители и асессоры назначены. Оценка еще не началась.
В процессе: Участники проходят оценку, руководители и асессоры выставляют баллы.
На проверке администратора: Все оценки выставлены. Система рассчитала результаты и направила их администратору для проверки.
На данном этапе администратор видит:
    • оценки асессоров;
    • оценки руководителя отдела;
    • оценки руководителя департамента;
    • итоговые значения по компетенциям;
    • рассчитанные группы обучения.
Администратор может:
    • подтвердить результаты;
    • вернуть оценки на пересмотр;
    • скорректировать состав групп обучения.
Подтверждена: Администратор подтвердил результаты ассессмента и сформированные группы обучения.
Опубликована: Результаты опубликованы.
После публикации:
    • результаты отображаются в профиле сотрудника;
    • данные становятся доступны для формирования программ развития;
    • данные становятся доступны для формирования ПИР.

Администратор
Может:
    • создавать кампании;
    • назначать участников;
    • назначать асессоров;
    • создавать и редактировать критерии;
    • устанавливать минимальные проходные баллы;
    • просматривать оценки всех участников;
    • подтверждать результаты ассессмента;
    • публиковать результаты;
    • запускать автоматическое формирование групп обучения;
    • изменять состав сформированных групп;
    • подтверждать группы обучения;
    • повторно запускать алгоритм группировки;
    • управлять справочником текстовых интерпретаций; 
    • создавать, изменять и удалять интерпретации; 
    • просматривать историю изменений справочника.
Руководитель отдела
Может:
    • видеть сотрудников своего отдела;
    • выставлять оценки по утвержденным критериям;
    • просматривать опубликованные результаты своих сотрудников;
    • просматривать автоматически сформированную интерпретацию; 
    • редактировать текст интерпретации перед сохранением оценки.
Не может:
    • видеть оценки других руководителей;
    • изменять итоговые результаты.
Руководитель департамента
Может:
    • видеть сотрудников своего департамента;
    • выставлять оценки по утвержденным критериям;
    • просматривать опубликованные результаты своих сотрудников;
    • просматривать автоматически сформированную интерпретацию; 
    • редактировать текст интерпретации перед сохранением оценки.
Не может:
    • изменять итоговые результаты;
    • изменять группы обучения.
Асессор
Может:
    • видеть только назначенных сотрудников;
    • выставлять оценки по назначенным компетенциям;
    • редактировать свои оценки до завершения кампании;
    • просматривать автоматически сформированную интерпретацию; 
    • редактировать текст интерпретации перед сохранением оценки.
Не может:
    • видеть оценки других асессоров;
    • видеть оценки руководителей;
    • видеть итоговые результаты до публикации.
AtS / Отдел развития
Может:
    • просматривать подтвержденные результаты ассессмента;
    • просматривать автоматически сформированные группы обучения;
    • видеть сильные стороны групп;
    • видеть зоны развития групп;
    • использовать данные для формирования программ обучения;
    • использовать данные для формирования ПИР.
Не может:
    • изменять оценки;
    • изменять результаты ассессмента.
Сотрудник
Может:
    • видеть опубликованные результаты;
    • видеть дату последнего ассессмента;
    • видеть итоговый профиль компетенций после публикации.
Не может:
    • изменять оценки;
    • видеть результаты до публикации;
    • видеть результаты других сотрудников.
    7. Критические правила   🟡 ЧАСТИЧНО
> Выполнено: дробные итоги (п.2), несколько асессоров (п.3), автоматический расчёт итога (п.4, см. AS8).
> Расхождение: шкала в БД 0–10, а не 1–10 (п.1).

    1. Все оценки выставляются по шкале от 1 до 10.
    2. Допускаются дробные итоговые значения.
    3. Один сотрудник может оцениваться несколькими асессорами.
    4. Итоговый профиль сотрудника рассчитывается автоматически системой после завершения кампании.
    5. Для автоматического формирования групп обучения используются только оценки асессоров.
    6. Оценки руководителя отдела и руководителя департамента используются как дополнительный источник информации для администратора и не участвуют в автоматической группировке сотрудников.
    7. До подтверждения администратором результаты не публикуются.
    8. До подтверждения администратором группы обучения не считаются окончательными.
    9. Все результаты ассессмента сохраняются в истории кампании.
    10. После публикации результаты становятся доступны для формирования ПИР.
    11. После публикации результаты становятся доступны AtS для разработки программ обучения.
    12. Изменение опубликованных результатов допускается только администратором.
    13. Все изменения состава групп обучения должны фиксироваться в журнале действий.
    14. Один сотрудник может входить только в одну группу обучения.
    15. Группы обучения формируются автоматически на основании ранжирования сотрудников по результатам ассессмента.
    16. Ранжирование выполняется по возрастанию итогового среднего балла.
    17. При анализе используются только опубликованные результаты ассессмента.
    18. Размер группы по умолчанию составляет 12 человек и может изменяться администратором.
    19. Система должна автоматически пересчитывать группы после каждого нового цикла ассессмента.
    20. Результаты группировки являются рекомендательными и могут быть скорректированы администратором вручную.
    21. При выборе оценки система должна автоматически отображать текстовую интерпретацию выбранного балла.
    22. Текстовая интерпретация определяется по связке: Департамент → Грейд → Компетенция → Балл.
    23. Пользователь, выставляющий оценку, может отредактировать автоматически предложенную интерпретацию перед сохранением оценки.
    24. Система должна сохранять одновременно:
        ◦ числовую оценку; 
        ◦ автоматически сформированную интерпретацию; 
        ◦ итоговую редакцию пользователя. 
    25. Изменение справочника текстовых интерпретаций не должно изменять ранее сохранённые результаты оценки сотрудников.
